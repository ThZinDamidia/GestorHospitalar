📖 Documentação de Estrutura e Lógica — HMS
Esta documentação detalha as escolhas técnicas, estruturas de dados e a lógica de negócio implementada na Fase 0 do Sistema de Gestão Hospitalar.

1. Arquitetura de Dados
O sistema utiliza uma abordagem de Persistência em Memória (In-memory storage). Os dados são estruturados em dicionários aninhados para garantir performance e integridade.

🗂️ Estruturas Utilizadas
Dicionários (dict): Utilizados como base de dados principal (_medicos e _pacientes).

Porquê? A busca por Chave-Valor permite acesso de tempo constante O(1). Ao pesquisar por um ID Médico ou NIF de Paciente, o sistema não precisa percorrer uma lista, indo diretamente ao endereço de memória do registo.

Sets (Conjuntos): Embora o armazenamento final seja em dicionários, a lógica de exclusividade em nomes e campos críticos é validada para evitar redundância, mantendo a integridade dos dados.

2. Modelagem das Entidades (Fase 0)
🩺 Entidade: Médico
Representa o corpo clínico. A sua chave primária é um ID Único gerado sequencialmente.

Dados Base: Nome, Data de Nascimento, Nacionalidade.

Dados Profissionais: Especialidade, Data de Registo, Cargo e Horário.

Análise Qualitativa: Inclusão de campos de "Ponto Forte" e "Ponto Fraco" para facilitar a gestão de equipas e alocação em turnos críticos.

👤 Entidade: Paciente
Representa o utente. A sua chave primária é o NIF, gerado internamente.

Histórico Clínico: Tipagem sanguínea, alergias, doenças crónicas e cirurgias anteriores.

Vínculo Sistémico: Cada paciente possui um campo id_medico, criando uma relação direta com a entidade Médico.

3. Lógica de Negócio e Algoritmos
🛡️ Validação de Integridade Referencial
O sistema impede a criação de "pacientes órfãos". Antes de concluir o registo de um paciente, a função medico_existe() verifica se o ID do médico responsável está ativo na base de dados de médicos.

🔢 Algoritmo de Geração de NIF
Diferente de um gerador aleatório simples, o sistema utiliza o Cálculo do Dígito de Controlo (Módulo 11):

Gera 8 dígitos aleatórios começando por 1 ou 2.

Calcula a soma ponderada desses dígitos.

Determina o dígito verificador para garantir que o NIF é matematicamente válido segundo o padrão oficial.

📅 Validação Cronológica
Utilização do módulo datetime para assegurar que:

Datas de nascimento e registo seguem o formato ISO (YYYY-MM-DD).

Entradas de texto inválidas em campos de data não causem o encerramento do programa (Exception Handling).

4. Evolução do Relacionamento (Roadmap)
Atualmente, o relacionamento é de 1 para N (Um médico é responsável por vários pacientes).

Próxima Implementação: Introdução da entidade CONSULTA.
Esta nova estrutura transformará a lógica atual numa relação N para N, permitindo que um paciente seja atendido por múltiplos especialistas e que cada interação gere um registo histórico único (ID_CONSULTA).

5. Simulação de Camada de Rede (Logs)
Para preparar o código para uma futura migração para Web/API, foi implementada a função log_servidor. Ela intercepta cada operação de escrita/leitura e emite um reporte formatado:

Erro 400: Quando o utilizador falha na lógica de input (ex: ID da unidade não é um número).

Erro 404: Quando o identificador solicitado não consta na memória.

Sucesso 200/201: Confirmação de persistência ou recuperação de dados.
